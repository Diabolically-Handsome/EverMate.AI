## ✨ 为什么是 EverMate.AI？
当今大多数在线 AI 都有两个痛点：  
1) **记忆长度有限**：聊到一定长度就要“清空重来”，之前的您被忘得一干二净。  
2) **隐私成谜**：聊天是否会被上传、训练、分析？您很难确认。

**EverMate.AI 的回答**：**默认本地存储与推理**，把“长期陪伴对话”做成**可用且可管**的产品体验。我们用两大核心能力让记忆既“短而有力”也“长而可靠”：

- **记忆三分法（Core → Persona → Vault）**：像人一样分层储存，既快又准。  
- **可扩展本地索引（Large-scale · Local · Fast）**：落盘分块 + 倒排索引 + BM25 检索，证据句就地抽取。

结果：常谈常新的高频主题“随手就有”，长尾往事“随叫随到”，且全程在您本机完成。

## 🍱 招牌功能（读者最关心的三件事）
### 1) 记忆三分法（Core → Persona → Vault）
- **Core · 核心记忆**：  
  高频主题 + AI 人设/回复模板线索（如：称呼用“您”、给例子、语气温和）。大多数问题直接命中，**响应更快更稳**。  
- **Persona · 用户画像**：  
  沟通偏好、信息密度、长期兴趣与禁忌等，**≤ 8 条要点**凝练；优先用**本地 LLM**精炼，不可用时启发式回退。  
- **Vault · 长尾记忆**：  
  其余细节按**分块落盘**并用 **BM25** 检索，**随用随取、短而准**。  
- **冲突处理**：当历史与当下输入冲突，以**当下为准**，避免被旧信息牵制。

### 2) 可扩展本地索引（Large-scale · Local · Fast）
- **分块写盘**：约 **2.8K 字符/块**（可调），适配百万～数百万字规模。  
- **倒排索引**：基于 SQLite，`terms/postings/chunks` 三表，WAL 模式读写稳健。  
- **BM25 检索**：命中相关块后，**从块内挑最佳原句**作为证据片段注入上下文。  
- **流式构建 & 增量更新**：拖拽导入 `.docx/.txt` 时边读边建；新建朋友**边聊边写**，达阈值自动刷新 Core/Persona。

### 3) “十秒钟上手”的交互心法  
**先 Core 定风格与高频主题** → **用 Persona 补偏好** → **从 Vault 取出处与证据句**，上下文**短而有力**。

**可选 · 画像更准：** 配置本地 **Ollama**（如 `OLLAMA_URL`、`OLLAMA_MODEL`），走本地 LLM 精炼。

## 💼 工作原理（文本示意图）
```
           ┌─────────┐    高频主题/风格线索     ┌─────────────┐
Input ───▶ │  Core   │ ───────────────────────▶ │  应答生成器 │
           └─────────┘                          └─────────────┘
                 ▲                                       ▲
                 │画像要点（≤8）                          │证据句
           ┌─────────┐                              ┌───────────┐
           │ Persona │ ◀──────── 本地LLM/启发式 ─── │  BM25检索 │
           └─────────┘                              └───────────┘
                                                        ▲
                                                        │ 命中块→最佳原句
                                                  ┌─────────────┐
                                                  │   Vault     │
                                                  │ 分块落盘索引 │
                                                  └─────────────┘
```
- **短路优先**：能被 Core/Persona 直击的，不必拉长上下文。  
- **证据注入**：需要“回忆”时，从命中块抽出**原话证据**注入，从而可验证、可追溯。

## 🧲 导入与新建（两条路，同一内核）
- **拖拽导入**：`.docx` 走标准库解析；`.txt` 流式读取。全量建索引后即用三分记忆应答。  
- **新建朋友**：`append_turn` 每回合增量入库，默认每 **20** 个新块刷新 Core/Persona。

---

## 🔧 可调参数（按需微调）
- `CHUNK_CHARS`：默认 **2800**（块大小）；**变大**更省索引，**变小**召回更细。  
- `CORE_TOP_TERMS`：默认 **50**（Core 展示的高频主题数）。  
- `PERSONA_MAX_BULLETS`：默认 **8**（画像要点上限）。  
- `REFRESH_EVERY`：默认 **20**（新增多少块后自动刷新 Core/Persona）。  
- 检索 **TopK**：`retrieve(query, k)`，建议 **4～8**。

**环境变量（示例）**
```bash
# 本地 LLM（可选）
export OLLAMA_URL="http://localhost:11434"
export OLLAMA_MODEL="qwen2.5:7b-instruct"   # 示例，按需替换

# 索引与内存目录
export MEMORY_DIR="./memory"
```

## ❓ 常见问题（围绕招牌功能）
- **画像还没变？**  
  可能未触发刷新阈值；继续聊天或手动重建，也可调小 `REFRESH_EVERY`。  
- **想看“原话证据”？**  
  可以，长尾检索会给出最相关的原句/片段。  
- **索引越来越大？**  
  调大 `CHUNK_CHARS`、阶段性归档旧 `chunks/`、或按“朋友”拆分独立 `memory` 目录。  
- **没有本地 LLM？**  
  只影响 Persona 凝练；算法有启发式回退，不中断流程。  
- **历史把我束缚住了？**  
  系统永远以**当前输入**为准；也可精修 `01_core.md / 02_persona.md` 后重建。
